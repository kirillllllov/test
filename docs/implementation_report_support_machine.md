# Отчет о реализации: Машина состояний поддержки и интеграция

Этот документ описывает процесс разработки и интеграции логики для работы агентов поддержки в групповом чате Telegram.

## 1. Введение и Постановка задачи
**Цель:** Обеспечить возможность сотрудникам поддержки обрабатывать обращения (Appeals) непосредственно в групповом чате Telegram.
**Проблема:** Отсутствовала логика управления состоянием обращения (кто взял в работу, на какой стадии находится решение) и механизм взаимодействия с ботом в контексте группы.

## 2. Реализация Машины Состояний (`supportAppealMachine`)

### Идея и Проектирование
Для управления жизненным циклом обращения была выбрана библиотека **XState**. Это позволяет четко определить возможные статусы и переходы, исключая некорректные состояния (например, нельзя закрыть обращение, которое никто не взял в работу).

**Ключевые решения:**
*   **Контекст машины**: Хранит `appealId`, ID и имя сотрудника (`accepterEmployee`), а также текст решения.
*   **Изоляция**: Каждое обращение — это отдельный экземпляр машины (Actor), что позволяет параллельно обрабатывать множество тикетов.

### Этапы разработки
1.  **Определение состояний**:
    *   `Created` (Новый): Начальное состояние. Ожидает события `TAKE_WORK`.
    *   `In_progress` (В работе): Обращение закреплено за сотрудником. Доступны действия `SOLVE` (решить), `REASSIGN` (переназначить), `RELEASE` (вернуть в очередь).
    *   `Solving` (Написание решения): Внутреннее состояние, когда бот ждет текст ответа от сотрудника.
    *   `Closed` (Закрыт): Финальное состояние.

2.  **Кодирование (`src/machines/support-appeal-machine.ts`)**:
    *   Описана структура `createMachine`.
    *   Реализованы `actions` (действия) для обновления контекста (например, `assignEmployee`).
    *   Добавлены заглушки для сайд-эффектов (логирование в консоль вместо реальных вызовов API).

3.  **Модульное тестирование**:
    *   Создан скрипт `src/scripts/test-support-machine.ts`.
    *   Проведена симуляция полного цикла: Взятие в работу -> Переход к решению -> Отправка ответа -> Закрытие.

## 3. Реализация Контроллера (`MainBotController`)

### Идея и Проектирование
Необходим единый входной шлюз для всех обновлений от Telegram (Webhooks), который умеет маршрутизировать сообщения.
**Сложность:** В групповом чате сообщения идут сплошным потоком. Нужно понимать, к какому именно обращению относится конкретное сообщение или нажатие кнопки.

**Решение:** Использовать ID обращения (`Appeal #123`) как ключ для маршрутизации и восстановления состояния.

### Этапы разработки
1.  **Создание контроллера (`src/controllers/main-bot-controller.ts`)**:
    *   Реализован метод `handleWebhook`: Разделяет входящие обновления на `callback_query` (кнопки) и `message` (текст).
    *   Реализован метод `handleSupportGroupMessage`:
        *   Анализирует текст сообщения или reply (ответ) на наличие паттерна `Appeal #ID`.
        *   Если ID найден, восстанавливает состояние машины через `stateService`.
    *   Реализована обработка кнопок (`handleCallbackQuery`): Транслирует нажатия кнопок (TAKE_WORK, SOLVE) в события машины состояний.

2.  **Управление состоянием (`stateService`)**:
    *   Использован существующий сервис (in-memory cache) для хранения активных машин состояний, привязанных к `appealId`.

3.  **Настройка маршрутизации (`src/routes.ts`)**:
    *   Добавлен POST-эндпоинт `/api/v1/webhook`, который принимает запросы от Telegram и передает их в контроллер.

## 4. Интеграционное Тестирование

Для проверки всей цепочки был создан скрипт `src/scripts/test-webhook.ts`, который эмулирует поведение Telegram-сервера.

**Сценарий теста:**
1.  **Эмуляция Callback `TAKE_WORK`**:
    *   Отправлен JSON с нажатием кнопки.
    *   **Результат**: Контроллер нашел машину, перевел её в статус `In_progress`, залогировал действие.
2.  **Эмуляция Callback `SOLVE`**:
    *   Отправлен JSON с нажатием кнопки "Решить".
    *   **Результат**: Машина перешла в состояние `Solving`.
3.  **Эмуляция Текстового ответа**:
    *   Отправлен JSON с текстом решения (как reply на тикет).
    *   **Результат**: Контроллер передал текст в машину, она сохранила решение и перешла в статус `Closed`.

## 5. Итог
Система успешно реализует базовый рабочий процесс поддержки. Архитектура (Controller -> Service -> FSM) соблюдена, что обеспечивает масштабируемость и удобство поддержки кода.
